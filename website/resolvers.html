Resolvers

Resolvers locate a source image based on the image identifier in a request URL. In Java-speak, they take in an identifier and return a File or ReadableByteChannel object from which the corresponding image can be read by a processor.

(This distinction is important because not all processors can work with all resolvers.)

The resolution process can be configured to be static — with all requests using the same resolver — or dynamic, with images being served from different resolvers depending on their identifier.

Static Resolver

By default, the resolver.static configuration key is set to FilesystemResolver, which tells Cantaloupe to use FilesystemResolver for all requests. This is simple and works in most scenarios.

Dynamic Resolvers

It is possible to serve images from different sources simultaneously. For example, you may have images some stored on a filesystem, and others stored on Amazon S3. If you can differentiate their sources based on their identifier in code — either by analyzing the identifier string, or performing some kind of service request — you can write a simple Ruby function to tell Cantaloupe which resolver to use for a given request.

The script can be named anything, but it must have a .rb extension. Then, it can be referenced by the resolver.chooser_script option, like:

# Option 1: absolute pathname
resolver.chooser_script = /path/to/functions.rb

# Option 2: filename only; will be expected to be located in the
# same folder as the config file, or the current working directory
resolver.chooser_script = functions.rb

This file must contain a module called Cantaloupe, and inside that, a function called get_resolver(identifier), which takes in an identifier and returns a resolver name (such as FilesystemResolver, HttpResolver, etc.) For example:

# functions.rb

module Cantaloupe
  def self.get_resolver(identifier)
    # Here, you would perform some kind of analysis on `identifier`:
    # analyze it, look it up in a web service or database...
    # and then return the name of the resolver to use to serve it.
    'FilesystemResolver'
  end
end

# Uncomment to test on the command line (`ruby functions.rb`)
# puts Cantaloupe::get_resolver('identifier-to-test')

FilesystemResolver

FilesystemResolver maps a URL identifier to a filesystem path, for retrieving images on a local or attached filesystem.

For images with extensions that are missing or unrecognized, this resolver will check the "magic number" to determine type, which will add some overhead. It is therefore slightly more efficient to serve images with extensions.

FilesystemResolver supports two distinct lookup strategies, defined by the FilesystemResolver.lookup_strategy configuration option.

BasicLookupStrategy

BasicLookupStrategy locates images by concatenating a pre-defined path prefix and/or suffix. For example, with the following configuration options set:

# Note trailing slash!
FilesystemResolver.BasicLookupStrategy.path_prefix = /usr/local/images/
FilesystemResolver.BasicLookupStrategy.path_suffix =

An identifier of image.jpg in the URL will resolve to /usr/local/images/image.jpg.

Note: it is dangerous to not use path_prefix. The shallower the path, the more of the filesystem that will be exposed.
To prevent arbitrary directory traversal, BasicLookupStrategy will strip out ..{path separator} and {path separator}.. from identifiers before resolving the path.

ScriptLookupStrategy

Sometimes, BasicLookupStrategy will not offer enough control. Perhaps you want to serve images from multiple filesystems, or perhaps your identifiers are opaque and you need to perform a database or web service request to locate the corresponding images. With this lookup strategy, you can tell FilesystemResolver to invoke your own custom Ruby function and capture the pathname it returns.

The script can be named anything, but it must have a .rb extension. Then, it can be referenced by the FilesystemResolver.ScriptLookupStrategy.script option, like:

# Option 1: absolute pathname
FilesystemResolver.ScriptLookupStrategy.script = /path/to/functions.rb

# Option 2: filename only; will be expected to be located in the
# same folder as the config file, or the current working directory
FilesystemResolver.ScriptLookupStrategy.script = functions.rb

This file must contain a module called Cantaloupe, and inside that, a function called get_pathname(identifier), which takes in an identifier and returns a pathname, if available, or nil, if not. A Solr query, for example, could look like:

# functions.rb
require 'net/http'
require 'cgi'

module Cantaloupe
  def self.get_pathname(identifier)
    uri = 'http://localhost:8983/solr/collection1/select?q=' +
        CGI.escape('id:"' + identifier + '"') +
        '&fl=pathname_si&wt=ruby'
    uri = URI.parse(uri)

    http = Net::HTTP.new(uri.host, uri.port)
    request = Net::HTTP::Get.new(uri.request_uri)
    response = http.request(request)
    return nil if response.code.to_i >= 400

    results = eval(response.body)['response']['docs']
    results.any? ? results.first['pathname_si'] : nil
  end
end

# Uncomment to test on the command line (`ruby functions.rb`)
# puts Cantaloupe::get_pathname('identifier-to-test')

Path Separator

The IIIF Image API specifications mandate that identifiers in URLs be percent-encoded, which changes slashes to %2F. Some reverse-proxy servers automatically decode this into / before passing on the request. This will cause HTTP 404 errors in Cantaloupe.

If you are unable to configure your reverse proxy to stop decoding slashes, the FilesystemResolver.path_separator configuration option will enable you to use an alternate string as a path separator in an identifier. (Be sure to supply the non-percent-encoded string, and then percent-encode it in URLs if it contains any URL-unsafe characters.)

HttpResolver

HttpResolver maps a URL identifier to an HTTP resource, for retrieving images from a web server.

It is preferable to use this resolver with source images with recognizable file extensions. For images with an extension that is missing or unrecognizable, it will issue an HTTP HEAD request to the server to check the Content-Type header. If the type cannot be inferred from that, an HTTP 415 response will be returned.

HttpResolver supports two distinct lookup strategies, defined by the HttpResolver.lookup_strategy configuration option.

BasicLookupStrategy

BasicLookupStrategy locates images by concatenating a pre-defined URL prefix and/or suffix. For example, with the following configuration options set:

# Note trailing slash!
HttpResolver.url_prefix = http://example.org/images/
HttpResolver.url_suffix =

An identifier of image.jpg in the URL will resolve to http://example.org/images/image.jpg.

ScriptLookupStrategy

Sometimes, BasicLookupStrategy will not offer enough control. Perhaps you want to serve images from multiple URLs, or perhaps your identifiers are opaque and you need to run a database or web service request to locate them. With this lookup strategy, you can tell HttpResolver to invoke your own custom Ruby script and capture the URL it returns.

The script can be named anything, but it must have a .rb extension. Then, it can be referenced by the HttpResolver.ScriptLookupStrategy.script option, like:

# Option 1: absolute pathname
HttpResolver.ScriptLookupStrategy.script = /path/to/functions.rb

# Option 2: filename only; will be expected to be located in the
# same folder as the config file, or the current working directory
HttpResolver.ScriptLookupStrategy.script = functions.rb

This file must contain a module called Cantaloupe, and inside that, a function called get_url(identifier), which takes in an identifier and returns a URL, if available, or nil, if not.

See Custom Scripts for general information about custom scripts in Cantaloupe, and the FilesystemResolver ScriptLookupStrategy section for an example of a similar script.

Path Separator

The IIIF Image API specifications mandate that identifiers in URLs be percent-encoded, which changes slashes to %2F. Some proxy servers automatically decode this into / before passing on the request. This will cause HTTP 404 errors in Cantaloupe.

If you are unable to configure your proxy to stop decoding slashes, the HttpResolver.path_separator configuration option will enable you to use an alternate string as a path separator in an identifier. (Be sure to supply the non-percent-encoded string, and then percent-encode it in URLs if it contains any URL-unsafe characters.)

JdbcResolver

JdbcResolver maps a URL identifier to an RDBMS BLOB field, for retrieving images from a relational database.

Cantaloupe does not include any JDBC drivers, so a driver JAR for the desired database must be obtained separately and saved somewhere on the classpath.

The JDBC connection is initialized by the JdbcResolver.connection_string, JdbcResolver.user, and JdbcResolver.password configuration options. If the user or password are not necessary, they can be left blank. The connection string must use your driver's JDBC syntax:

jdbc:postgresql://localhost:5432/my_database
jdbc:mysql://localhost:3306/my_database
jdbc:microsoft:sqlserver://example.org:1433;DatabaseName=MY_DATABASE

Consult the driver's documentation for details.

Then, the resolver needs to be told:

  1. The database value corresponding to a given identifier
  2. The media type corresponding to that value
  3. The SQL command that retrieves the BLOB corresponding to that value

Database Identifier Retrieval Function

This JavaScript function takes in an unencoded identifier and returns the corresponding database value of the identifier. For example, the following will work when the identifiers in the URL map directly to values in the database:

JdbcResolver.function.identifier = function getDatabaseIdentifier(url_identifier) {\
      return url_identifier;\
  }

Otherwise, if the URL identifiers require some transformation first:

JdbcResolver.function.identifier = function getDatabaseIdentifier(url_identifier) {\
        // Assumes that the identifier is stored in the database as\
        // "identifier-something"\
        return url_identifier + "-something";\
    }
Note that in properties files, all lines in multi-line strings except the last need to end with a backslash (\).

Media Type Retrieval Function

This JavaScript function returns a media (MIME) type corresponding to the value returned by the getDatabaseIdentifier() function. If the media type is stored in the database, this example will return an SQL statement to retrieve it.

JdbcResolver.function.media_type = function getMediaType(identifier) {\
      return "SELECT media_type " +\
          "FROM some_table " +\
          "WHERE some_identifier = ?"\
  }

If the URL identifier will always have a known, valid image extension, like .jpg, .tif, etc., the entire JdbcResolver.function.media_type option can be commented out, and Cantaloupe will infer the media type from the extension.

BLOB Retrieval SQL

JdbcResolver.lookup_sql is an SQL statement that selects the BLOB corresponding to the value returned by the getDatabaseIdentifier() function.

JdbcResolver.lookup_sql = SELECT image_blob_column \
    FROM some_table \
    WHERE some_identifier = ?

AmazonS3Resolver

AmazonS3Resolver maps a URL identifier to an Amazon S3 object. The identifier is passed as-is to S3.

AmazonS3Resolver currently only supports images with recognizable filename extensions. For images with extensions that are missing or unrecognized, an HTTP 404 error will be returned.

AmazonS3Resolver can be configured with the following options:

AmazonS3Resolver.access_key_id
An access key associated with your AWS account. (See AWS Security Credentials.)
AmazonS3Resolver.secret_key
A secret key associated with your AWS account. (See AWS Security Credentials.)
AmazonS3Resolver.bucket.name
Name of the bucket containing the images to be served.
AmazonS3Resolver.bucket.region
Name of a region to send requests to, such as us-east-1. Can be commented out or left blank to use a default region. (See S3 Regions.)
